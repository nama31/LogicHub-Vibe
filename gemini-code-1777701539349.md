## Phase 11: Route Data Model (Database & Schemas)
*   **Goal:** Expand the database schema to support grouped orders (routes) and sequentially track stops[cite: 6].
*   **Ideal Agent:** Claude 3.5 Sonnet (via Cursor Composer)
*   **Required Context:** `architecture.md` (Sections 4 & 8), `logihub/backend/models/order.py`, `logihub/backend/core/database.py`[cite: 6].
*   **READY PROMPT:**
> Act as a Senior Database Architect. Your task is to implement the new "Route" data model for LogiHub.
> STRICT RULES:
> 1. Create a new `routes` table in `logihub/backend/models/route.py` with: id (UUID), courier_id (FK), created_by (FK), label (TEXT), status (draft, active, completed, cancelled), started_at, completed_at, and created_at[cite: 6].
> 2. Update the existing `orders` table to include `route_id` (UUID, FK, nullable) and `stop_sequence` (INTEGER, nullable)[cite: 6].
> 3. Generate the corresponding Pydantic schemas in `logihub/backend/schemas/route.py`[cite: 6].
> 4. Run Alembic to autogenerate the migration file (e.g., `0005_create_routes.py`)[cite: 6].
> Do not write the API routers or business logic yet. Focus strictly on the data layer.
*   **Acceptance Criteria:** Alembic migration runs successfully. The database schema now reflects routes and the two new columns in the orders table[cite: 6].

## Phase 12: Route Backend Logic
*   **Goal:** Implement the API endpoints to manage route lifecycles and strictly enforce stop-by-stop sequencing[cite: 6].
*   **Ideal Agent:** Claude 3.5 Sonnet
*   **Required Context:** `logihub/backend/routers/routes.py`, `logihub/backend/services/route_service.py`, `logihub/backend/schemas/route.py`[cite: 6].
*   **READY PROMPT:**
> Act as a Senior Backend Engineer. Your task is to implement the business logic and FastAPI endpoints for Route management.
> STRICT RULES:
> 1. Implement CRUD operations: `POST /routes`, `GET /routes`, `GET /routes/:id`, `PATCH /routes/:id`, and `DELETE /routes/:id` (cancels draft and returns orders to 'new')[cite: 6].
> 2. Implement `POST /routes/:id/start`. This sets the route to 'active' and triggers a Telegram notification to the courier[cite: 6].
> 3. Implement `PATCH /bot/routes/:id/stop/:stop_id/complete` (internal bot endpoint)[cite: 6].
> 4. CRITICAL: Enforce atomic DB transactions. Only ONE stop per route can be `in_transit` at a time[cite: 6]. When a stop is marked complete, the system must automatically transition the next sequential order from 'assigned' to 'in_transit'[cite: 6].
> Maintain single responsibility. Do not touch frontend code.
*   **Acceptance Criteria:** You can successfully create a route, group multiple unassigned orders into it, and hit the start/stop endpoints via Swagger (`/docs`) without breaking state[cite: 6].
